# パフォーマンス・セキュリティレビュー

変更されたファイルを対象に、パフォーマンスとセキュリティの観点でレビューし、問題があれば修正する。

## 手順

### 1. 変更箇所の特定

```bash
git diff --name-only HEAD
```

変更のあった `.rs` ファイルを特定する。

### 2. パフォーマンスレビュー

変更箇所を Read で読み、以下をチェック:

| チェック項目 | 問題パターン | 修正方針 |
|---|---|---|
| ループ内ヒープ確保 | `Vec::new()`, `.to_vec()`, `format!()`, `String::new()`, `.collect()` | スタック配列、事前確保、キャッシュ |
| COM オブジェクト毎フレーム生成 | `CreateSolidColorBrush`, `CreateTextLayout` がループ内 | `HashMap` でキャッシュ |
| Mutex 二重ロック | 同じ Mutex を同一メソッド内で2回以上 `.lock()` | 1回のロックで必要な値をまとめて取得 |
| 不要な clone/copy | `.clone()` が回避可能 | 参照で保持、ロック保持中に操作 |
| O(n) の頻繁呼出 | 毎フレーム/毎コールバックで全グリッド走査 | 軽量な代替メソッドを用意 |

### 3. セキュリティレビュー

変更箇所を Read で読み、以下をチェック:

| チェック項目 | 問題パターン | 修正方針 |
|---|---|---|
| unsafe ポインタ操作 | `from_raw_parts`, `copy_nonoverlapping`, `*ptr.add(n)` | null チェック、境界チェック (`GlobalSize` 等) |
| 整数キャスト | `as i32`, `as u16`, `as usize` | `saturating_add/sub`、`try_from`、範囲チェック |
| 外部入力のバッファ | クリップボード、PTY、IME からのデータ | サイズ上限の検証 |
| Mutex `.unwrap()` | panic でロック毒殺の連鎖 | クリティカルパスでは `unwrap_or_else` を検討 |
| ハンドル/ポインタの寿命 | スレッド間での HWND/HANDLE 共有                      | 所有権モデルの確認                     |
| エラーの握りつぶし       | `?` → `unwrap_or_default()` / `ok()` / `unwrap_or()` | 根本原因を調査し、原因そのものを修正する |
| COM 初期化の連鎖失敗     | `setup_xxx().ok()` で COM セットアップ失敗が無視される、`AdviseSink` 等の `?` で後続ステップが全て無効化 | 各ステップの成功を個別に検証。部分失敗を許容する場合は `match` で明示的にハンドル |

### 4. 問題の修正

発見した問題を重要度順に修正する。修正後は `cargo build --release` で確認。

### 5. レポート

修正内容を箇条書きで報告する。問題がなければ「問題なし」と報告。

## 制約

- 変更のあったファイルのみを対象とする（プロジェクト全体をスキャンしない）
- 軽微な問題（1回限りの呼出での format! 等）は無視してよい
- 修正は最小限に。既存の動作を変えない
