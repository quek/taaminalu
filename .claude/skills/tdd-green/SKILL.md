# TDD Green フェーズ — テストを通す実装を書く

失敗しているテストを通すために、最小限の実装コードを書く。

テストは正しい仕様を表現している。実装がテストに合わせる。テストを変えるのではない。

## 手順

### 1. 失敗テストの確認

```bash
cargo test 2>&1
```

現在失敗しているテストを特定し、以下を把握する:

- どのテストが失敗しているか（テスト名）
- 失敗の種類（アサーション失敗の内容）
- テストが期待しているインターフェースと振る舞い

テストコードを Read で読み、期待される振る舞いを正確に理解する。

### 2. 実装

テストが通るように実装コードを書く。

ガイドライン:
- テストが要求する最小限の実装を書く（YAGNI）
- 既存コードの設計・命名規則に合わせる
- 新しい関数・構造体を追加する場合は、適切なモジュールに配置する

### 3. テスト通過の確認

```bash
cargo test
```

**全テストが成功すること**を確認する。

- 新規テストだけでなく、既存テストも壊れていないこと
- 失敗するテストがある場合は実装を修正する（テストは修正しない）

### 4. コミット

全テスト成功を確認したら、実装ファイルのみをコミットする。

```bash
git add <実装ファイル>
git commit -m "[Green] ○○を実装"
```

- コミットメッセージは日本語
- プレフィックスは `[Green]`
- テストファイルはステージングしない（変更していないはずだが、念のため確認）

## テストが間違っていると判断した場合

実装を進める中で「このテスト自体が間違っている」と判断した場合、
自分でテストを修正してはならない。代わりに Red エージェントに判断を委ねる。

Agent ツールで以下の内容を Red エージェントに送る:

- 対象テスト名
- 元のユーザー要求（テストが書かれた背景・意図）
- テストのどこが間違っていると考えるか（具体的な根拠）
- 実装で判明した事実（テスト作成時には分からなかった制約等）
- 正しいと考える期待値・振る舞い

Red エージェントからの返答に従う:

| Red の判断               | Green の対応                             |
|--------------------------|------------------------------------------|
| 「テストは正しい」       | 実装を修正して、テストを通す             |
| 「ユーザーに確認を求める」 | 作業を中断し、ユーザーの判断を待つ       |

## 禁止事項（厳守）

- **テストコードを絶対に修正してはならない**
  - `#[cfg(test)]` モジュール内のコードを編集しない
  - `tests/` ディレクトリ内のファイルを編集しない
  - テストのアサーション値を変えない
  - `#[ignore]` を追加しない
  - テストと実装が同一ファイルにある場合（例: `src/term.rs` 内の `#[cfg(test)] mod tests`）、
    `#[cfg(test)]` ブロックの外側（実装部分）は編集してよい
- テストが間違っていると思っても、自分で修正せず Red エージェントに報告する
