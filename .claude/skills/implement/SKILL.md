# 機能実装ワークフロー

$ARGUMENTS を実装する。

調査→統合テスト→実装→確認の順で進める。
テストはリグレッション防止を目的とし、可能な限り高いレイヤーで書く。

## 手順

### 1. 類似プロダクトの調査

**推測で実装するな。** まず正しい振る舞いを調査してから実装する。

以下のいずれかに該当する場合、`/research-similar-impl` スキルを呼び出して
類似プロダクト（Alacritty, WezTerm, Windows Terminal 等）の実装を調査する:

| 該当条件                                                       | 例                                       |
|----------------------------------------------------------------|------------------------------------------|
| 「○○みたいに」「○○と同じように」と参考プロダクトが指定されている | 「WezTerm みたいにダブルクリックで選択」 |
| ターミナルエミュレータとして一般的な機能である                   | 単語選択、URL 検出、括弧対応、スクロール |
| 正しい振る舞いが VT 仕様や慣習に依存する                       | エスケープシーケンスの処理、キーバインド |

調査で以下を明らかにする:
- **類似プロダクトの実際の振る舞い**（どういう入力に対してどう動くか）
- **エッジケースの扱い**（行末、空行、特殊文字等でどう動くか）
- **実装上の設計判断**（どういうアルゴリズム・データ構造を使っているか）

該当しない場合（内部リファクタリング、単純なバグ修正等）はスキップしてよい。

### 2. 要件の整理

調査結果と既存コード（Read/Grep で確認）をもとに、要件を整理する。

以下の観点で要件を箇条書きにする:

| 観点                   | 問いかけ                                                             |
|------------------------|----------------------------------------------------------------------|
| 正常系                 | 基本的な入力に対して何を返す／何が起きるべきか？                     |
| エッジケース           | 空入力、画面端、全角文字、スクロールバック中など                     |
| 既存機能との相互作用   | 既存の機能に影響しないか？組み合わせで壊れるケースはないか？         |
| 類似プロダクトとの一致 | 調査した振る舞いをすべてカバーしているか？                           |

**要件一覧をユーザーに提示し、過不足がないか確認を取る。**
ユーザーの承認を得てから次のステップに進む。

### 3. 統合テストの作成

承認された要件をもとに、統合テストを書く。

#### テストのレイヤー

可能な限り高いレイヤーでテストする。上から順に検討し、最も高いレイヤーを選ぶ。

| レイヤー               | 方法                                                                  | 例                                           |
|------------------------|----------------------------------------------------------------------|----------------------------------------------|
| **Win32 メッセージ**   | `SendMessage` で WM_* を送信し、結果の状態を検証                     | ダブルクリック選択、キー入力、スクロール       |
| **グリッド操作**       | `TermWrapper` に VT シーケンスをフィードし、メソッド呼び出し結果を検証 | 単語境界検出、テキスト抽出、ACP 変換          |
| **純粋ロジック**       | 関数に入力を与え、出力を検証                                         | スクロール行数計算、座標変換                   |

Win32 メッセージレベルのテストが最もリグレッション検出力が高い。
セットアップが重くても、テストヘルパーにまとめて簡潔に書けるようにする。

#### テスト設計のガイドライン

- **1 テスト = 1 つのユーザーシナリオ**
- 期待値は `assert_eq!` で具体的な値を検証する（`starts_with()` や `> 0` は使わない）
- 単純な入出力パターンはパラメタライズドテストにまとめる:

```rust
#[test]
fn ダブルクリックで各種単語が選択される() {
    let cases = [
        ("hello world",  col: 2, "hello"),
        ("hello world",  col: 7, "world"),
        ("foo-bar baz",  col: 1, "foo"),
        ("日本語テスト", col: 0, "日本語テスト"),
    ];
    for (text, col, expected) in cases {
        let app = TestApp::new(80, 24);
        app.feed_text(text);
        app.send_double_click(0, col);
        assert_eq!(app.selected_text(), expected, "text={text:?}, col={col}");
    }
}
```

- 自明な初期値テスト（`assert_eq!(x.field(), 0)`）は書かない
- テストヘルパー（`TestApp` 等）を積極的に作り、Arrange を簡潔に保つ

#### コンパイルを通す

テスト対象の関数・構造体がまだ存在しない場合は、コンパイルが通るための最小限のスタブを追加してよい。
スタブはデフォルト値を返す空実装とする。

#### テスト失敗の確認

```bash
cargo test
```

- コンパイルが通ること
- 新規テストがアサーション失敗で落ちること（テストが意味のある検証をしている証拠）
- 既存テストは壊れていないこと

### 4. 実装

テストが通るように実装する。

ガイドライン:
- 既存コードの設計・命名規則に合わせる
- KISS・DRY を意識する

### 5. 全テスト通過の確認

```bash
cargo test
```

**新規テスト・既存テスト全てが通ること**を確認する。
失敗するテストがあれば実装を修正する。

### 6. リファクタリング（必要に応じて）

全テストが通った状態で、コードを整理する。

OK: リネーム、関数抽出、重複排除、`clippy` 警告修正、テストヘルパー整理
NG: 新機能追加（次のサイクルでやる）

リファクタリング後も全テストが通ることを確認する。

### 7. コミット

```bash
cargo test
cargo clippy -- -D warnings
git add <変更ファイル>
git commit -m "○○を実装"
```

- コミットメッセージは日本語
- テストと実装を 1 コミットにまとめる
- コンパイル警告を残さない

## テストが間違っていると気づいた場合

実装中にテストの期待値が間違っていると判断した場合:

1. 根拠を明確にする（調査結果、実際の動作、仕様書等）
2. ユーザーに報告し、テスト修正の承認を得る
3. 承認後にテストを修正し、実装を続ける

## 禁止事項

- 推測で実装しない（調査してから実装する）
- `#[ignore]` でテストをスキップしない
- ユーザーの承認なしにテストの期待値を変更しない
