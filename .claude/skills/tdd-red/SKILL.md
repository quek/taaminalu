# TDD Red フェーズ — 失敗するテストを書く

$ARGUMENTS に対するテストを書き、失敗を確認してコミットする。

テストが先、実装は後。このフェーズではテストコードだけを書く。
テストが失敗する状態（Red）をコミットすることで、テストが意味のある検証をしていることを証明する。

## 手順

### 1. 類似プロダクトの調査（テスト設計の前に必ず実施）

**推測でテストを書くな。** まず正しい振る舞いを調査してからテストを書く。

ユーザーの機能要求が以下のいずれかに該当する場合、`/research-similar-impl` スキルを呼び出して
類似プロダクト（Alacritty, WezTerm, Windows Terminal 等）の実装を調査する:

| 該当条件                                               | 例                                       |
|--------------------------------------------------------|------------------------------------------|
| 「○○みたいに」「○○と同じように」と参考プロダクトが指定されている | 「WezTerm みたいにダブルクリックで選択」 |
| ターミナルエミュレータとして一般的な機能である                   | 単語選択、URL 検出、括弧対応、スクロール |
| 正しい振る舞いが VT 仕様や慣習に依存する                       | エスケープシーケンスの処理、キーバインド |

調査で以下を明らかにする:
- **類似プロダクトの実際の振る舞い**（どういう入力に対してどう動くか）
- **エッジケースの扱い**（行末、空行、特殊文字等でどう動くか）
- **実装上の設計判断**（どういうアルゴリズム・データ構造を使っているか）

### 2. 要件の洗い出し

調査結果と既存コード（Read/Grep で確認）をもとに、要件を網羅的に洗い出す。
実装もれの主な原因は、Red フェーズでテストに落とし込まれなかった要件である。
このステップで要件を一覧化し、ユーザーに確認することで漏れを防ぐ。

以下の観点で要件を箇条書きにする:

| 観点                   | 問いかけ                                                             |
|------------------------|----------------------------------------------------------------------|
| 正常系                 | 基本的な入力に対して何を返す／何が起きるべきか？                     |
| 境界値                 | 空文字列、0、最大値、配列の先頭/末尾、画面端など                     |
| 異常系                 | 不正な入力、範囲外のインデックス、null/None に相当するケース         |
| 副作用                 | 状態変更、イベント発火、PTY への送信など                             |
| 既存機能との相互作用   | 既存のテスト・機能に影響しないか？組み合わせで壊れるケースはないか？ |
| 類似プロダクトとの一致 | 調査した振る舞いをすべてカバーしているか？見落としはないか？         |

**洗い出した要件一覧をユーザーに提示し、過不足がないか確認を取る。**
特に類似プロダクトの調査結果と照らし合わせて、漏れがないか確認する。
ユーザーの承認を得てから次のステップに進む。

### 3. テスト設計

承認された要件一覧をもとに、以下を決定する:

- テスト対象の関数・メソッドのシグネチャ
- 各要件に対応するテストケース
- テストの配置場所（同一ファイルの `#[cfg(test)]` モジュール、または `tests/` ディレクトリ）

各要件がテストケースに対応していることを確認する。
要件一覧に対応するテストがないものがあれば、テストケースを追加する。

#### テスト粒度のガイドライン

**「1つの判断ロジック = 1テスト」** を基本とする。
テストの労力は機能の複雑さに比例させる。単純な計算に多くのテストを費やし、
複雑な機能のテストが薄くなる「労力の逆転」を避ける。

| パターン               | 方針                                                       | 例                                             |
|------------------------|----------------------------------------------------------|------------------------------------------------|
| 単純な入出力対応       | パラメタライズドテストで 1 つにまとめる                   | `calc_scroll_lines` の各 delta 値              |
| 判断ロジック           | 分岐ごとに 1 テスト                                      | 単語境界の検出（英単語、日本語、記号）         |
| 複雑な機能             | 正常系・境界値・組み合わせを手厚くテスト                 | 選択テキスト抽出（複数行、全角、行末トリム）   |
| 自明な初期値           | 書かない（他のテストの Arrange で暗黙的にカバーされる）   | `初期状態の○○は0`                              |

**書くべきでないテスト:**
- 自明な初期値の確認（`assert_eq!(x.default_field(), 0)`）
- 内部状態の増減方向だけの検証（`assert!(x > 0)` — 具体的な期待値を書く）
- `starts_with()` 等による部分検証（期待値全体を検証する）

**パラメタライズドテストの書き方:**

```rust
#[test]
fn test_calc_scroll_lines_各パターン() {
    let cases = [
        (120, 3, 3),    // 1ノッチ上、3行/ノッチ → 3行
        (-120, 3, -3),  // 1ノッチ下 → -3行
        (240, 3, 6),    // 2ノッチ上 → 6行
        (60, 3, 1),     // 半ノッチ → 最低1行
        (120, 5, 5),    // lines_per_notch=5
    ];
    for (delta, lines_per_notch, expected) in cases {
        assert_eq!(
            calc_scroll_lines(delta, lines_per_notch), expected,
            "delta={delta}, lines_per_notch={lines_per_notch}"
        );
    }
}
```

### 4. テストコードの作成

テストコードを書く。

**コンパイルは必ず通るようにする。** テスト対象の関数・構造体がまだ存在しない場合は、
コンパイルが通るための最小限のスタブを実装側に追加してよい。
スタブはデフォルト値を返す空実装（`String::new()`, `0`, `false`, `Vec::new()` 等）とする。
`todo!()` や `unimplemented!()` は使わない（panic はアサーション失敗ではないため）。
テストはアサーション失敗（not コンパイルエラー、not panic）で落ちること。

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_機能名_正常系() {
        // Arrange
        // Act
        // Assert
    }
}
```

ガイドライン:
- Arrange-Act-Assert パターンで構造化する
- テスト名は日本語でもよい（`test_単語選択_英単語`）
- 1 テストにつき 1 つの振る舞いを検証する
- 境界値・エッジケースも忘れずに
- 期待値は具体的な値で検証する（`starts_with()` や `> 0` ではなく `assert_eq!` で厳密に）

### 5. 失敗の確認

```bash
cargo test
```

以下を確認する:

- **コンパイルが通ること**（エラーがあればスタブを追加して解消する）
- **テストがアサーション失敗で落ちること**（Red 状態）
- **テストが成功してしまった場合** → テストが不十分。アサーションを見直して、
  実装されていない機能を本当に検証しているか確認し、修正する

### 6. コミット

テスト失敗（アサーション失敗）を確認したら、変更ファイルをコミットする。

```bash
git add <テストファイル> <スタブを追加した実装ファイル（あれば）>
git commit -m "[Red] ○○のテストを追加"
```

- コミットメッセージは日本語
- プレフィックスは `[Red]`

## Green エージェントからテスト誤り報告を受けた場合

Green エージェントから「テストが間違っている」という報告を受けることがある。
テストの責任者として、報告内容を精査し判断する。

### 判断手順

1. 報告されたテストコードを Read で読む
2. テストが検証しようとしている仕様・振る舞いを再確認する
3. Green の指摘（期待値が間違い、前提条件が不正等）が妥当か検証する

### 判断結果と対応

| 判断                       | 対応                                                                   |
|----------------------------|------------------------------------------------------------------------|
| **テストは正しい**（Green の指摘が間違い） | Green に「テストは正しい。実装を修正せよ」と返答する                    |
| **テストが間違っている可能性がある**       | ユーザーに状況を説明し、テスト修正の可否について確認を求める           |

- Red が独断でテストを修正してはならない。テスト修正にはユーザーの承認が必要
- ユーザーが修正を承認した場合のみ、テストを修正してコミットする（`[Red] テスト修正: ○○`）

## Refactor エージェントからテスト削除報告を受けた場合

設計変更により不要になったテストについて、Refactor エージェントから削除報告を受けることがある。
テストの責任者として、報告内容を精査し判断する。

### 判断手順

1. 削除候補のテストコードを Read で読む
2. テストが検証している旧 API が本体コードから完全に削除されているか確認する
3. 新設計のテストが旧テストのカバレッジを代替しているか確認する

### 判断結果と対応

| 判断                             | 対応                                                                         |
|----------------------------------|------------------------------------------------------------------------------|
| **削除は妥当**                   | ユーザーに削除対象と理由を報告し、承認を求める                               |
| **削除すべきでない**             | Refactor に「テストは残すべき」と返答し、理由を説明する                       |
| **判断が難しい**                 | ユーザーに状況を説明し、判断を委ねる                                         |

- Red が独断でテストを削除してはならない。テスト削除にはユーザーの承認が必要
- ユーザーが削除を承認した場合、テストを削除し `cargo test` で全テストパス＆警告なしを確認してコミットする
- コミットメッセージ: `[Red] 不要テスト削除: ○○`

## 禁止事項

- **テストを通すロジックを実装してはならない**。スタブは「コンパイルを通す」ためだけ
- `#[ignore]` でテストをスキップしない
- ユーザーの承認なしにテストを修正・削除しない
