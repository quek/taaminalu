# TDD Red フェーズ — 失敗するテストを書く

$ARGUMENTS に対するテストを書き、失敗を確認してコミットする。

テストが先、実装は後。このフェーズではテストコードだけを書く。
テストが失敗する状態（Red）をコミットすることで、テストが意味のある検証をしていることを証明する。

## 手順

### 1. 要求の理解と設計

ユーザーの機能要求を分析し、以下を決定する:

- テスト対象の関数・メソッドのシグネチャ
- テストケース（正常系・境界値・異常系）
- テストの配置場所（同一ファイルの `#[cfg(test)]` モジュール、または `tests/` ディレクトリ）

既存コードを Read/Grep で確認し、テスト対象のモジュール構造を把握する。

### 2. テストコードの作成

テストコードを書く。

**コンパイルは必ず通るようにする。** テスト対象の関数・構造体がまだ存在しない場合は、
コンパイルが通るための最小限のスタブを実装側に追加してよい。
スタブはデフォルト値を返す空実装（`String::new()`, `0`, `false`, `Vec::new()` 等）とする。
`todo!()` や `unimplemented!()` は使わない（panic はアサーション失敗ではないため）。
テストはアサーション失敗（not コンパイルエラー、not panic）で落ちること。

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_機能名_正常系() {
        // Arrange
        // Act
        // Assert
    }
}
```

ガイドライン:
- Arrange-Act-Assert パターンで構造化する
- テスト名は日本語でもよい（`test_単語選択_英単語`）
- 1 テストにつき 1 つの振る舞いを検証する
- 境界値・エッジケースも忘れずに

### 3. 失敗の確認

```bash
cargo test
```

以下を確認する:

- **コンパイルが通ること**（エラーがあればスタブを追加して解消する）
- **テストがアサーション失敗で落ちること**（Red 状態）
- **テストが成功してしまった場合** → テストが不十分。アサーションを見直して、
  実装されていない機能を本当に検証しているか確認し、修正する

### 4. コミット

テスト失敗（アサーション失敗）を確認したら、変更ファイルをコミットする。

```bash
git add <テストファイル> <スタブを追加した実装ファイル（あれば）>
git commit -m "[Red] ○○のテストを追加"
```

- コミットメッセージは日本語
- プレフィックスは `[Red]`

## Green エージェントからテスト誤り報告を受けた場合

Green エージェントから「テストが間違っている」という報告を受けることがある。
テストの責任者として、報告内容を精査し判断する。

### 判断手順

1. 報告されたテストコードを Read で読む
2. テストが検証しようとしている仕様・振る舞いを再確認する
3. Green の指摘（期待値が間違い、前提条件が不正等）が妥当か検証する

### 判断結果と対応

| 判断                       | 対応                                                                   |
|----------------------------|------------------------------------------------------------------------|
| **テストは正しい**（Green の指摘が間違い） | Green に「テストは正しい。実装を修正せよ」と返答する                    |
| **テストが間違っている可能性がある**       | ユーザーに状況を説明し、テスト修正の可否について確認を求める           |

- Red が独断でテストを修正してはならない。テスト修正にはユーザーの承認が必要
- ユーザーが修正を承認した場合のみ、テストを修正してコミットする（`[Red] テスト修正: ○○`）

## Refactor エージェントからテスト削除報告を受けた場合

設計変更により不要になったテストについて、Refactor エージェントから削除報告を受けることがある。
テストの責任者として、報告内容を精査し判断する。

### 判断手順

1. 削除候補のテストコードを Read で読む
2. テストが検証している旧 API が本体コードから完全に削除されているか確認する
3. 新設計のテストが旧テストのカバレッジを代替しているか確認する

### 判断結果と対応

| 判断                             | 対応                                                                         |
|----------------------------------|------------------------------------------------------------------------------|
| **削除は妥当**                   | ユーザーに削除対象と理由を報告し、承認を求める                               |
| **削除すべきでない**             | Refactor に「テストは残すべき」と返答し、理由を説明する                       |
| **判断が難しい**                 | ユーザーに状況を説明し、判断を委ねる                                         |

- Red が独断でテストを削除してはならない。テスト削除にはユーザーの承認が必要
- ユーザーが削除を承認した場合、テストを削除し `cargo test` で全テストパス＆警告なしを確認してコミットする
- コミットメッセージ: `[Red] 不要テスト削除: ○○`

## 禁止事項

- **テストを通すロジックを実装してはならない**。スタブは「コンパイルを通す」ためだけ
- `#[ignore]` でテストをスキップしない
- ユーザーの承認なしにテストを修正・削除しない
