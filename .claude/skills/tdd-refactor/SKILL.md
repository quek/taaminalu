# TDD Refactor フェーズ — テストを Green に保ったままコードを整理する

Green フェーズで通ったテストを壊さずに、実装コードとテストコードの両方を整理する。

振る舞いは変えない。変えるのは構造・可読性・DRY だけ。

## 手順

### 1. 全テスト Green の確認

```bash
cargo test
```

全テストが通っていることを確認する。通っていなければ Refactor に入らない。

### 2. リファクタリング対象の特定

直近の Red/Green で変更されたファイルを確認する:

```bash
git diff --name-only HEAD~2..HEAD
```

以下の観点でリファクタリング対象を洗い出す:

| 観点             | 例                                                             |
|------------------|----------------------------------------------------------------|
| 重複コード       | 同じロジックが複数箇所にある → 関数に抽出                      |
| 命名             | 意図が伝わらない変数名・関数名 → より明確な名前に変更          |
| 構造             | 関数が長すぎる → 分割。責務が混在 → モジュール分離             |
| テストの可読性   | Arrange が長い → ヘルパー関数に抽出。マジックナンバー → 定数化 |

### 3. リファクタリングの実施

実装コード・テストコードの両方を編集してよい。
ただし、振る舞い（テストが検証する入出力の関係）は変えない。

OK な変更:
- リネーム（変数、関数、モジュール）
- 関数の抽出・インライン化
- 重複排除（DRY）
- テストヘルパーの追加
- コメントの整理
- `clippy` 警告の修正

NG な変更:
- 新しい機能の追加
- テストの期待値の変更
- テストの削除
- ロジックの変更（バグ修正含む。それは次の Red-Green サイクルでやる）

### 4. テスト Green の再確認

```bash
cargo test
```

**全テストが引き続き通ること**を確認する。
失敗した場合はリファクタリングで振る舞いを壊してしまったので、修正する。

### 5. コミット

```bash
git add <変更ファイル>
git commit -m "[Refactor] ○○を整理"
```

- コミットメッセージは日本語
- プレフィックスは `[Refactor]`

## 不要なテストを発見した場合

設計変更により本体コードから削除された旧 API のテストが残っている場合、
Refactor エージェントはテストを削除してはならない。代わりに Red エージェントに報告する。

Agent ツールで以下の内容を Red エージェントに送る:

- 削除候補のテスト名一覧
- 不要と判断した理由（旧 API が本体から削除済み、新設計のテストが代替している等）
- 残すべきテスト一覧（新設計をカバーしているもの）

Red エージェントが精査し、ユーザーの承認を得た上でテストを削除する。

## 禁止事項

- テストの期待値・アサーションを変更しない（振る舞いを変えない）
- 新機能を追加しない（それは次の Red フェーズでやる）
- テストを削除しない（不要テストは Red エージェントに報告する）
